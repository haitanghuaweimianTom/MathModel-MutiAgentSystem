# 数学建模论文自动生成系统 —— 技术说明文档

> 本文档详细说明本系统在架构设计、模块实现上借鉴了 **LLM-MM-Agent** 与 **Cherry Studio** 的哪些技术，以及我们在此基础上**自主设计**的重要创新部分。

---

## 一、概述

本项目（数学建模论文自动生成系统 v2.1）是一个面向数学建模竞赛（MCM/ICM/高教社杯等）的全自动论文生成流水线。系统以赛题文本和 Excel 数据文件为输入，自动完成问题分析、数学建模、算法设计、代码求解、结果分析与论文撰写，最终输出可直接提交的 Markdown + Word 格式论文。

在架构层面，我们**融合**了 LLM-MM-Agent 的数学建模专用工作流与 Cherry Studio 的通用 AI Agent 基础设施，并在此基础上进行了大量**面向数学建模场景的自主设计与深度改造**。

---

## 二、借鉴 LLM-MM-Agent 的核心技术

LLM-MM-Agent（Large Language Model for Mathematical Modeling Agent）是面向数学建模任务的多智能体框架。我们在以下方面直接借鉴并吸收了其设计思想：

### 2.1 Coordinator + DAG 调度与黑板内存（Blackboard Memory）

**借鉴点**：
- LLM-MM-Agent 采用 **Coordinator（中央调度器）** 统一管理多任务依赖关系，使用 **DAG（有向无环图）** 描述子任务间的数据/方法/结构依赖。
- 引入 **黑板内存（Blackboard Memory）** 机制，作为全局共享的上下文空间，各 Agent 可读写中间结果，避免重复计算。

**我们的实现**（`src/workflow/coordinator.py`）：
- 实现了 `Coordinator` 类，维护 `TaskNode` 任务节点集合。
- 支持 5 种依赖类型：`DATA`（数据依赖）、`METHODOLOGICAL`（方法依赖）、`COMPUTATIONAL`（计算依赖）、`STRUCTURAL`（结构依赖）、`CODE`（代码依赖）。
- 使用 **Kahn 算法** 计算 DAG 拓扑排序，确保任务按正确顺序执行。
- 每个任务节点维护 `memory`（结果黑板）和 `code_memory`（代码结构黑板），下游任务可通过 `get_dependency_context()` 自动拼接上游依赖的上下文。

### 2.2 CritiqueEngine：Actor-Critic-Improvement 质量保障

**借鉴点**：
- LLM-MM-Agent 提出 **多维批判框架**，对生成内容从多个维度（准确性、创新性、完整性、可解性等）进行批判性评估，并基于评估结果生成改进版本。
- 核心设计原则：**批判时从多个维度拆解评估；改进时禁止提及先前版本的缺陷，直接给出新版本**。

**我们的实现**（`src/workflow/critique_engine.py`）：
- 实现了完整的 `CritiqueEngine` 类，支持 4 种内容类型的批判维度：
  - `analysis`（问题分析）：思考深度、视角新颖性、逻辑严谨性、上下文意识、结构化程度
  - `modeling`（数学建模）：准确性与严谨性、创新与洞察、实际适用性、完整性、可解性
  - `algorithm`（算法设计）：正确性、效率、鲁棒性、可实现性
  - `paper_chapter`（论文章节）：内容充实度、逻辑连贯性、数据准确性、学术规范性、深度分析
- 每个维度给出 1-10 分评分 + 具体评论 + 可操作改进建议。
- 支持循环迭代：当综合评分低于阈值（默认 8.0）时，自动触发改进，最多迭代 2 轮。
- 改进 Prompt 严格遵守 LLM-MM-Agent 的约束：不提及原版本缺陷，直接输出改进后的完整内容。

### 2.3 四阶段工作流架构

**借鉴点**：
- LLM-MM-Agent 将数学建模任务划分为 4 个核心阶段：**Problem Analysis → Mathematical Modeling → Computational Solving → Paper Generation**。
- 各阶段之间通过显式接口传递结构化结果，而非自由文本。

**我们的实现**（`src/agent_workflow.py`）：
- 在 `UnifiedWorkflow` 中完整复现了这一四阶段架构：
  - **Stage 1**：问题分析（Problem Analysis）+ DAG 构建
  - **Stage 2**：数学建模（Mathematical Modeling）—— 按 DAG 拓扑序逐任务建模
  - **Stage 3**：计算求解（Computational Solving）—— 算法设计 + 代码生成 + 自动执行
  - **Stage 4**：论文生成（Paper Generation）—— 大纲驱动 + 相关性过滤

### 2.4 PaperGenerator：大纲驱动的论文章节生成

**借鉴点**：
- LLM-MM-Agent 的 `PaperGenerator` 采用 **OutlineGenerator + ContextExtractor + ContentGenerator** 三级结构。
- 通过**相关性映射（Relevance Mapping）**控制每章的上下文输入，避免将全部历史内容堆入 Prompt 导致上下文溢出。

**我们的实现**（`src/workflow/paper_generator.py`）：
- 实现了 `PaperGenerator` 类，职责包括：
  - 根据模板大纲逐章生成内容
  - 利用 `template.get_relevance_context()` 控制上下文长度（`max_chars=5000`）
  - 每章生成后进行字数检查与 Critique-Improvement
  - 组装完整论文

### 2.5 coding_actor + coding_debugger 设计

**借鉴点**：
- LLM-MM-Agent 将代码生成拆分为 **coding_actor**（负责编写代码）和 **coding_debugger**（负责修复错误），形成生成-调试闭环。

**我们的实现**（`src/workflow/code_executor.py`）：
- 实现了 `CodeExecutor` 类：
  - **代码生成**：调用 Claude CLI 或 API 生成 Python 求解代码
  - **隔离执行**：在 subprocess 沙箱中运行代码，捕获 stdout/stderr
  - **自动修复**：执行失败时，将错误信息反馈给 LLM，要求修复代码后重试（最多 4 轮）
  - **结果提取**：自动提取执行结果并保存为结构化 JSON

### 2.6 HMML（Hierarchical Mathematical Modeling Library）分层方法知识库

**借鉴点**：
- LLM-MM-Agent 构建了分层数学建模方法库，按 **Domain → Subdomain → Method** 三级组织，支持基于问题描述的方法检索与推荐。

**我们的实现**（`src/agents/coordinator.py`）：
- 实现了 `MethodKnowledgeBase` 类，包含：
  - `MethodDomain`：光谱学、干涉、优化、回归、信号处理等领域枚举
  - `MethodNode`：方法节点，包含公式、参数、假设、应用场景、相关方法链
  - `suggest_methods()`：基于关键词匹配为问题推荐相关方法
  - `get_method_chain()`：获取方法关联链，支持方法组合推荐

### 2.7 反思链设计（Chain-of-Thought / Reflection）

**借鉴点**：
- LLM-MM-Agent 在问题分析阶段采用 **反思链设计（Reflection Chain）**，要求模型从多个视角（数学、物理、工程）审视问题，识别关键假设和不确定性。

**我们的实现**（`src/agent_workflow.py` → `_stage_problem_analysis`）：
- 在问题分析 Prompt 中明确要求：
  1. 识别核心组件及依赖关系
  2. 分析动态特性（时间演化、空间分布）
  3. 从数学、物理、工程多视角审视
  4. 识别关键假设和潜在不确定性
  5. 将问题分解为可管理的子任务

## 三、借鉴 Cherry Studio 的核心技术

Cherry Studio 是一个通用的 AI Agent 桌面客户端，我们在其开源架构基础上，借鉴了其 Provider 抽象、Agent 管理、RAG 知识库、MCP 工具层等通用基础设施设计。

### 3.1 Provider 架构设计 —— 多 LLM 提供商统一抽象

**借鉴点**：
- Cherry Studio 的 **Provider 架构** 通过统一的抽象接口屏蔽不同 LLM API 的差异（OpenAI、Anthropic、Gemini、Ollama 等）。
- 支持从环境变量自动读取配置，实现零配置启动。

**我们的实现**（`src/llm/base.py`、`src/llm/factory.py`）：
- 定义了 `BaseLLMProvider` 抽象基类，统一接口 `generate()`。
- 实现了 `ProviderType` 枚举：OPENAI、ANTHROPIC、GEMINI、OLLAMA、CLAUDE_CLI。
- `ProviderConfig` 支持从环境变量自动初始化（`OPENAI_API_KEY`、`ANTHROPIC_API_KEY` 等）。
- `LLMProviderFactory` 实现工厂模式，根据配置动态创建对应 Provider 实例。
- **扩展**：新增了 `CLAUDE_CLI` Provider，将 Claude Code CLI 作为正式 Provider 纳入统一调度。

### 3.2 Agent 管理基类与注册表

**借鉴点**：
- Cherry Studio 的 **AgentService** 架构，通过 `AgentEntity` 定义 Agent 的可配置属性（ID、名称、角色、系统提示词、模型、能力列表等）。
- 使用注册表模式管理所有 Agent 实例。

**我们的实现**（`src/agents/manager/`）：
- `AgentConfig`（借鉴 `AgentEntity`）：定义了完整的 Agent 配置结构，包括：
  - `role`：COORDINATOR、PROBLEM_ANALYZER、MODEL_DESIGNER、ALGORITHM_DESIGNER、CODE_WRITER、PAPER_WRITER 等
  - `model` / `plan_model` / `small_model`：支持多模型分工（主模型用于生成，轻量模型用于简单任务）
  - `capabilities`：Agent 能力列表
  - `accessible_paths`：Agent 可访问的文件路径（权限隔离）
  - `tools`：可用工具列表
- `AgentRegistry`：注册表管理，支持按角色检索 Agent
- `AgentFactory`：工厂模式，根据配置动态创建 Agent 实例

### 3.3 KnowledgeBase：RAG 知识增强

**借鉴点**：
- Cherry Studio 的 **KnowledgeService** 设计，封装了完整的 RAG（Retrieval-Augmented Generation）流程：文档分块 → 嵌入 → 向量存储 → 语义检索。
- 支持多种嵌入模型（Sentence-Transformer、TF-IDF 等）。

**我们的实现**（`src/knowledge/`）：
- `KnowledgeBase` 类（借鉴 `KnowledgeService`）：
  - `DocumentChunker`：文档分块，支持 `chunk_size` 和 `chunk_overlap` 配置
  - `EmbeddingModel` 抽象接口，`SentenceTransformerEmbedding` 和 `TfidfEmbedding` 两种实现
  - `VectorStore`：向量存储与相似度检索
  - `add_document()` / `query_with_context()`：文档添加与上下文检索
- **应用场景**：在数学建模阶段，自动检索与问题描述相关的数学方法，增强建模的准确性和多样性。

### 3.4 DocumentLoader：多格式文档处理

**借鉴点**：
- Cherry Studio 的文档预处理模块，支持多种文件格式的加载与文本提取。

**我们的实现**（`src/document_processing/`）：
- 实现了 `DocumentLoader` 类，支持：
  - Excel 文件（`.xlsx`）读取：自动提取列名、数据类型、统计摘要
  - Markdown 文件读取
  - 文本摘要生成：为数据文件生成 1500 字以内的结构化描述
- 数据文件描述自动注入到问题分析 Prompt 中，确保 LLM 了解数据结构和字段含义。

### 3.5 MCP Hub：MCP 工具管理

**借鉴点**：
- Cherry Studio 的 **MCP Hub** 设计，统一管理多个 MCP（Model Context Protocol）服务器的工具发现与调用。

**我们的实现**（`src/mcp/`）：
- `MCPToolManager` 类（借鉴 `MCP Hub`）：
  - `connect_stdio()`：通过 stdio 连接 MCP 服务器
  - `connect_from_config()`：从配置字典连接
  - 自动工具发现：连接后自动扫描服务器提供的工具列表
  - `ToolInfo` 数据结构：统一管理工具的名称、描述、输入模式、启用状态

### 3.6 Mermaid 可视化支持

**借鉴点**：
- Cherry Studio 内置的 Mermaid 图表渲染支持。

**我们的实现**（`src/visualization/mermaid/`）：
- 实现了 Mermaid 图表生成器，用于自动生成工作流 DAG 图、数学建模流程图等可视化内容。

## 四、自主设计的重要部分

在借鉴上述两个开源项目的基础上，我们针对**数学建模竞赛论文生成**这一特定场景，进行了大量自主设计与深度改造。以下是我们**独立设计并实现**的核心创新：

### 4.1 显式记忆池（Explicit Memory Pool）—— 跨阶段上下文传递

**问题背景**：
- 原生 LLM-MM-Agent 各阶段间传递的是截断后的文本摘要（如 `[:1000]`、`[:2000]`），导致模型/算法/代码/结果四者严重脱节。
- 论文生成阶段如果直接塞入全部原始内容，Prompt 会溢出；如果截断，又会丢失关键信息。

**我们的设计**（`src/agent_workflow.py`）：
- 在 `UnifiedWorkflow` 中维护一个全局 `memory_pool` 字典，包含 5 类结构化摘要：
  - `analysis_summary`（400-500 字）：问题背景、关键子问题、核心假设、解决思路、关键数值约束
  - `modeling_summary`（400-500 字）：核心变量、核心公式、求解策略、子问题映射
  - `algorithm_summary`（200-300 字）：算法名称、输入数据、核心步骤、输出格式
  - `results_summary`（400-500 字）：关键数值结果表格、主要发现、与模型的对应关系、可视化建议
  - `chapter_summaries`（200-300 字/章）：每章生成后自动提炼的结构化摘要
- 每阶段完成后，调用 LLM 将原始结果**压缩**为固定格式的结构化摘要，供后续阶段显式调用。
- **优势**：既避免了上下文溢出，又确保了跨阶段信息不丢失、逻辑连贯。

### 4.2 分段逐章生成 + 预生成大纲 —— 论文质量保障机制

**问题背景**：
- 一次性生成整篇论文（20000+ 字）会导致 LLM 后期遗忘前期内容，各章风格不一致、数据矛盾。
- 没有大纲约束时，LLM 容易偏离主题或遗漏关键章节。

**我们的设计**（`src/workflow/paper_generator.py`）：
- **预生成大纲**：正式写章前，先批量生成各章详细大纲，明确每章必须覆盖的要点和字数目标。
- **逐章生成**：每章独立调用 LLM，Prompt 中明确包含：
  - 本章大纲（必须覆盖的要点）
  - 相关阶段摘要（从 memory_pool 提取）
  - 前 2 章的摘要（确保衔接自然）
- **字数保障**：每章生成后统计中文字符数，若低于 `min_chars`，自动触发扩展机制（调用 LLM 增加推导过程、分析讨论和实例说明）。
- **内容净化层**（`_sanitize_chapter_content`）：自动检测并过滤 LLM 输出的题目原文、重复标题等污染内容。

### 4.3 代码执行闭环 —— 从"幻觉代码"到"可验证结果"

**问题背景**：
- 传统 Agent 系统生成的代码往往包含中文前缀、markdown 标记、未定义的变量，执行失败率高达 90% 以上。
- 即使代码能运行，也常常因为结果为空或全为 0，导致论文中的数值结果全靠 LLM 幻觉编造。

**我们的设计**（`src/workflow/code_executor.py`）：
- **强制纯净代码**：Prompt 中明确要求"代码开头必须是 import 语句，不要有任何中文说明文字"，并内置 `_extract_code()` 自动去除 markdown 代码块标记。
- **结果契约**：要求代码运行后必须将结构化结果写入指定 JSON 文件（`results.json`），格式严格统一。
- **隔离执行**：在 subprocess 沙箱中运行代码，设置 60 秒超时，避免死循环或资源耗尽。
- **自动修复循环**：执行失败时，将 stderr 错误信息反馈给 LLM，要求修复后重试，最多 4 轮。
- **结果验证**：检查 `results.json` 是否存在、数值是否非空、是否在合理范围内。

### 4.4 多模板系统 —— 一套引擎支持多种论文类型

**问题背景**：
- 数学建模竞赛、课程作业、金融分析报告三种场景，论文结构和写作规范差异很大。
- 传统系统通常只能生成单一类型的论文。

**我们的设计**（`src/workflow/templates.py`）：
- 抽象基类 `PaperTemplate`，定义统一的模板接口：
  - `get_outline()`：获取论文大纲（章节列表）
  - `get_system_prompt()`：获取该类型的系统提示词
  - `get_relevance_context()`：根据章节相关性映射提取上下文
- 实现了 3 种具体模板：
  - `MathModelingTemplate`（12 章）：摘要、问题重述、问题分析、模型假设、符号说明、模型建立、模型求解、结果分析、灵敏度分析、模型评价与改进、参考文献、附录
  - `CourseworkTemplate`（8 章）：面向一般课程作业
  - `FinancialAnalysisTemplate`（10 章）：面向金融数据分析与投资报告
- 每章通过 `ChapterSpec` 定义目标字数、相关性键、是否依赖代码/数据等元信息。

### 4.5 Provider Fallback 机制 —— 高可用 LLM 调用链

**问题背景**：
- 单一 API 提供商可能因网络、额度、限速等原因不可用，导致整个流水线中断。
- Claude Code CLI 虽然稳定，但调用成本高、响应慢。

**我们的设计**（`src/agent_workflow.py` → `_call_llm`）：
- **优先级队列**：优先使用 API Provider（OpenAI → Anthropic → Gemini → Ollama），最后回退到 Claude CLI。
- **自动重试**：每次调用支持 3 次重试，间隔 5 秒。
- **故障转移**：API Provider 连续失败后，自动打印日志并切换到 Claude CLI。
- **统一接口**：无论底层使用哪种 Provider，对外暴露统一的 `_call_llm(prompt, system_prompt)` 接口，上层业务无感知。

### 4.6 图表自动生成 —— 从"文字描述图"到"真实 PNG 文件"

**问题背景**：
- 传统系统的"图表生成"只是在论文中用文字描述"如图 X 所示"，实际上没有任何图片文件，导致论文中的图表引用均为虚假引用。

**我们的设计**（`src/agent_workflow.py` → `_generate_charts`）：
- **数值数据提取**：递归扫描 `execution_result` 和 `results.json`，提取所有数值型结果。
- **智能图表策略**：
  - 策略 1：如果有多个数值指标，生成对比柱状图（`fig_01_comparison.png`）
  - 策略 2：如果结果是按任务组织的，生成饼图（`fig_02_pie.png`）
  - 策略 3：如果没有任何数值结果，生成占位提示图
- **高分辨率输出**：所有图表保存为 300 DPI 的 PNG，确保打印清晰度。
- **中文字体支持**：自动配置 matplotlib 中文字体（Noto Sans CJK SC / SimHei / DejaVu Sans）。

### 4.7 双格式导出 —— Markdown + Word 并行输出

**问题背景**：
- 数学建模竞赛最终提交物通常为 Word 文档（.docx），但 Markdown 便于人工审阅和版本控制。
- 传统系统只输出 Markdown，需要用户手动转换。

**我们的设计**（`src/agent_workflow.py` → `_convert_to_docx`）：
- **双格式并行**：论文生成后，同时保存 Markdown 文件（`MathModeling_Paper.md`）和 Word 文件（`数学建模论文.docx`）。
- **Pandoc 优先**：优先调用 pandoc 进行转换（质量最好、公式保留最完整）。
- **python-docx 回退**：若 pandoc 未安装，使用 python-docx 手动转换，自动处理：
  - 标题层级（# → 标题，## → 一级标题，### → 二级标题）
  - 表格解析与创建
  - 图片嵌入（自动提取 `![](path)` 并插入文档）
  - 中文字体设置（宋体正文、黑体标题）

### 4.8 统一工作流引擎 v2.1 —— 从"并行架构"到"单一流水线"

**问题背景**：
- 早期版本（v1.0）存在两套并行架构：`agent_workflow.py`（LLM 驱动层）和 `framework.py`/`workflow.py`（模板驱动层），导致代码重复、维护困难。

**我们的设计**（`src/agent_workflow.py` → `UnifiedWorkflow`）：
- **统一单一流水线**：将所有阶段整合到 `UnifiedWorkflow` 一个类中，消除架构分裂。
- **显式阶段接口**：每个 Stage 都有明确的输入输出和摘要生成逻辑：
  - `_stage_problem_analysis()` → `analysis_summary`
  - `_stage_mathematical_modeling()` → `modeling_summary`
  - `_stage_computational_solving()` → `algorithm_summary` + `results_summary`
  - `_stage_paper_generation_v2()` → `chapter_summaries`
- **文件引用式上下文**：各阶段间不传递长文本，而是传递文件路径，LLM 通过文件读取获取完整上下文（在 Prompt 中嵌入文件路径或摘要）。

## 六、结语

本项目并非简单地将 LLM-MM-Agent 和 Cherry Studio 拼接在一起，而是在深入理解两者设计思想的基础上，针对**数学建模竞赛论文生成**这一垂直场景进行了大量**场景化改造和自主创新**：

- **从"通用 Agent"到"专用流水线"**：将通用多 Agent 框架改造为 8 阶段数学建模专用流水线。
- **从"文本传递"到"记忆池传递"**：用结构化摘要替代文本截断，解决了长链路上下文丢失问题。
- **从"幻觉生成"到"可验证生成"**：通过代码执行闭环和结果契约，确保论文中的每个数值都有真实来源。
- **从"单一输出"到"双格式交付"**：同时输出 Markdown（便于审阅）和 Word（可直接提交）。

这些自主设计使得系统从"能生成一段文字"升级为"能生成可直接提交的完整竞赛论文"。
